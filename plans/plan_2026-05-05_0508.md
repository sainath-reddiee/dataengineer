### CRITICAL BUGS

#### 1. `ReadabilityPage.jsx:34` — Direct WP API fetch (SPA shell issue)
```js
const res = await fetch(`https://app.dataengineerhub.blog/wp-json/wp/v2/posts?slug=${selected}&_fields=content`);
```
This fetches from WordPress API but does NOT use `fetchBlogArticleHTML`. While it gets raw WP content (which works), it's inconsistent with the pattern used everywhere else. The real issue: `content.rendered` from WP API returns only the post body, NOT a full HTML doc with `<head>` etc. The `analyzeReadability()` function expects this (it strips HTML), so it works — but this page lacks any `fetchBlogArticleHTML` integration for consistency and could break if the API endpoint changes.

**Severity:** Low (functional but inconsistent)

#### 2. `FreshnessPage.jsx:140` — Broken Edit link
```js
href={`https://app.dataengineerhub.blog/wp-admin/post.php?action=edit&post=${article.slug}`}
```
The `post` parameter in WordPress admin requires a **numeric post ID**, not a slug. This link will always fail to load the correct post for editing.

**Severity:** Medium (broken UI functionality)

#### 3. `ContentOptimizerService.js:587` — Infinite recursion risk
```js

```
If `this.currentProxyIndex` cycles back to 0, the function throws. But if there's a timing issue or the proxy count changes, this could infinitely recurse. There's no maximum retry counter.

**Severity:** Low (edge case, unlikely in practice)

#### 4. `ScannerPage.jsx:26-31` — React Hook dependency warning / stale closure
```js
useEffect(() => {
    const urlParam = searchParams.get('url');
    if (urlParam && !report) {
        setUrl(urlParam);

```
`handleScan` is not in the dependency array and `report` is read from a closure that may be stale on re-renders. This is a React linting warning and can cause the scan to trigger more than once or miss triggering.
